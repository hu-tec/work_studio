## Part 1. 시스템 전체 구조

### 3분류

| | AI Studio | Work Studio | 별개 사이트 |
|---|-----------|-------------|-------------|
| **한마디** | 직원용 도구 | 데이터 창고 + 플러그인 공장 | 소비자용 웹사이트 |
| **사용자** | 사내 직원, 관리자 | (직접 안 씀 — 뒤에서 데이터 처리) | 외부 고객, 지원자, 학생 |
| **역할** | 출퇴근, 업무일지, 면접, 규정 등 내부 업무 | 신청서 접수 데이터 저장, 폼 엔진 제공, 시험 결과 저장 | 홈페이지, 신청페이지, 시험 응시 등 |
| **기술** | React + TypeScript + Express | Express + SQLite (vanilla JS) | 사이트마다 다름 |
| **경로** | `_ALL_CODES/ai_studio/` | `_ALL_CODES/work_studio/` | 각각 독립 |

### 판단 트리 — 새 기능을 어디에 만들지?

```
Q1. 직원이 내부 업무용으로 쓰는 건가?
    → YES: AI Studio
    → NO: 다음 질문

Q2. 소비자/외부인의 접수·신청·응시 데이터를 다루는 건가?
    → YES: Work Studio (DB/API)
    → NO: 다음 질문

Q3. 자체 브랜딩·자체 화면·자체 결제가 있는 소비자 사이트인가?
    → YES: 별개 사이트 (백엔드는 Work Studio API 사용)
```

**핵심: "누가 쓰느냐"가 아니라 "어떤 데이터를 다루느냐"**

| 상황 | 판단 | 이유 |
|------|------|------|
| 관리자가 직원 출퇴근 데이터를 봄 | AI Studio | 데이터 주체 = 직원 |
| 관리자가 소비자 신청서를 봄 | Work Studio | 데이터 주체 = 소비자 |
| 관리자가 외부 전문가 지원 현황을 봄 | Work Studio | 데이터 주체 = 외부 전문가 |
| 관리자가 내부 규정을 편집함 | AI Studio | 데이터 주체 = 회사 내부 |

## Part 2. 분류표 (A~E 5축)

> 회사의 모든 콘텐츠(시험문제, 교재, 신청서, 커리큘럼...)를 **5가지 기준**으로 정리하는 체계.

| 기준 | 이름 | 질문 | 예시 |
|------|------|------|------|
| **A** | 소재 분류 | "어떤 소재/도메인을 다루나?" | 법률 문서, 유튜브 영상, 요리 전문 |
| **B** | 시험 급수 | "어떤 자격 종목의 몇 급?" | 번역 비즈니스 3급, 프롬프트 전문 2급 |
| **C** | 홈페이지 | "어느 사이트에서 쓰나?" | TESOL 교육, AI 번역, 시험사이트 |
| **D** | 담당 조직 | "어느 팀이 관리하나?" | 문제은행팀-AI번역, 강사팀-윤리 |
| **E** | 역할/권한 | "누가 어떤 동작을 할 수 있나?" | 강사=편집가능, 알바=조회만, 외부=계약범위만 |

### A. 소재 분류 — "어떤 소재/도메인?"

| 대분류 | 성격 | 중분류 → 소분류 |
|--------|------|-----------------|
| **문서** | 콘텐츠 형태 | 비즈니스(사업계획서, 회사소개, PPT, 엑셀, 기획서), 법률(소송장, 준비서면, 형사, 민사), 의료, 특허, 노무, 교재, 논문, 기사, 고전, 그외 |
| **음성** | 콘텐츠 형태 | 아나운서, 관광가이드, 큐레이터, 안내방송, 교육강의, 실시간, 화상수업 |
| **영상/SNS** | 콘텐츠 형태 | 미디어/장르(유튜브, 다큐멘터리, 영화, 드라마, 예능) |
| **IT/개발** | 콘텐츠 형태 | 개발/보안(AI, 에이전트, DB, 빅데이터, 백엔드, 프론트, 프로그램), 디자인/기획(웹디자인, 웹기획, UIUX, 컨텐츠) |
| **창의적활동** | 콘텐츠 형태 | 콘텐츠(드라마, 웹툰소설, 소설, 시, 음악, 미술) |
| **번역** | 작업 방법론 | 통번역 방식(순차통역, 동시통역, 음성번역, 자가선택) |
| **프롬프트** | 작업 방법론 | -- |
| **확장영역** | 도메인 전문성 | 라이프/전문(요리, 재무, 주식, 부동산, 자녀, 연애, 입시, 사주, 결혼, 영어, 직장찾기, 운동, 사업, 암) |

> **확장영역**: 도메인 스페셜리스트를 분류하기 위한 것. "번역 x 확장>요리" = 요리 도메인 전문 번역가.

### B. 시험 급수 — 5개 종목, 3구분, 최대 13단계

**5대 자격시험 종목**: TESOL, AI번역, AI프롬프트, AI윤리, ITT정통번역

| 구분 | 급수 범위 | 수준 |
|------|-----------|------|
| **교육** | 1급 → 8급 | 입문 ~ 교육 최고 |
| **비즈니스** | 1급 → 3급 | 교육보다 상위 |
| **전문** | 1급 → 2급 | 최상위 |

5개 종목 모두 동일한 급수 체계를 사용한다.

**레벨 순서 (낮음 → 높음):**
```
교육 1급(입문) → ... → 교육 8급 → 비즈니스 1급 → 비즈니스 2급 → 비즈니스 3급 → 전문 1급 → 전문 2급(최상위)
```

> **시험 공통**: 5개 종목에 공통 적용되는 입학절차, 시험 비교표, 수속 안내. 별도 시험이 아니라 공통 섹션.

### E. 역할 & 권한 (RBAC)

E열은 "등급"이 아니라 **역할(Role)**이다. 일직선 서열이 아니라 유형별 권한이 다르다.

**역할 유형:**

| 유형 | 역할 | 설명 |
|------|------|------|
| 내부 비정규 | **알바** | 단순 보조 업무 |
| 내부 정규 | **신입** | 정규 입사, 소속 팀 업무 |
| 내부 전문 | **강사** | 종목별 교육/채점 담당 |
| 내부 관리 | **팀장** | 팀 관리 + 하위 팀원 감독 |
| 기술 | **개발** | 시스템 구축/유지보수 (조직 서열과 무관) |
| 외부 | **외부** | 외부 전문가/컨설턴트 (계약 범위만 접근) |
| 경영 | **임원** | 전사 데이터 열람 + 승인 |
| 최고경영 | **대표** | 최종 결재 + 시스템 설정 |

**권한 매트릭스:**

| 역할 | 조회 | 입력 | 편집 | 삭제 | 승인 | 시스템설정 | 데이터 범위 |
|------|------|------|------|------|------|-----------|------------|
| 알바 | ○ | — | — | — | — | — | 배정된 업무만 |
| 신입 | ○ | ○ | — | — | — | — | 소속 팀 테이블 |
| 강사 | ○ | ○ | ○ | — | — | — | 소속 종목 전체 (D팀 연동) |
| 팀장 | ○ | ○ | ○ | ○ | — | — | 소속 팀 전체 + 팀원 관리 |
| 개발 | ○ | ○ | ○ | ○ | — | ○ | 전체 (기술 운영 목적) |
| 외부 | ○ | ○ | — | — | — | — | 계약/프로젝트 범위만 |
| 임원 | ○ | ○ | ○ | ○ | ○ | — | 전체 데이터 |
| 대표 | ○ | ○ | ○ | ○ | ○ | ○ | 전체 + 최종 결재 |

**D그룹과 연동:**
- 강사의 "소속 종목" = D팀에서 결정 (예: 강사팀-AI번역 → AI번역 관련 테이블만)
- 팀장의 "소속 팀" = D팀 전체 (예: 문제은행팀 팀장 → 프롬프트+AI번역+윤리 문제 전체)
- 외부의 "계약 범위" = 별도 설정 (프로젝트별 테이블 지정)

## Part 3. 교차 활용

### 교차 1: 종목 x 소재 — "이 종목에서 어떤 소재를 다루나?"

| | 문서-법률 | 음성-강의 | 영상-유튜브 | 확장-요리 | 확장-재무 |
|---|---|---|---|---|---|
| **TESOL** | 법률 영어 교육 | 영어 강의 | 영어 교육 영상 | 요리 영어 | 재무 영어 |
| **AI번역** | 법률 번역 | 강의 통역 | 자막 번역 | 요리책 전문 번역 | 재무보고서 번역 |
| **AI프롬프트** | 법률문서 프롬프트 | 강의 요약 프롬프트 | 유튜브 스크립트 생성 | 요리 레시피 프롬프트 | 재무분석 프롬프트 |
| **AI윤리** | 법률 AI윤리 | -- | 미디어 윤리 | 식품 AI윤리 | 금융 AI윤리 |
| **ITT정통번역** | 법률 통번역 | 강의 통번역 | 영상 통번역 | 요리 통번역 | 재무 통번역 |

### 교차 2: 종목 x 급수 — "이 종목의 이 레벨"

| | 교육 1급(입문) | 교육 8급(교육최고) | 비즈니스 3급(비즈니스최고) | 전문 2급(최상위) |
|---|---|---|---|---|
| **TESOL** | TESOL 기초 | TESOL 교육 마스터 | TESOL 비즈니스 전문가 | TESOL 최고 전문가 |
| **AI번역** | 번역 기초 | 번역 교육 마스터 | 번역 비즈니스 전문가 | 번역 최고 전문가 |
| **AI프롬프트** | 프롬프트 기초 | 프롬프트 교육 마스터 | 프롬프트 비즈니스 전문가 | 프롬프트 최고 전문가 |
| **AI윤리** | 윤리 기초 | 윤리 교육 마스터 | 윤리 비즈니스 전문가 | 윤리 최고 전문가 |
| **ITT정통번역** | 통번역 기초 | 통번역 교육 마스터 | 통번역 비즈니스 전문가 | 통번역 최고 전문가 |

### 실제 태그 예시

```
시험 문제 #1247                         전문가 프로필 #89
├─ A소재:   문서 > 법률 > 소송장         ├─ A소재:   확장영역 > 라이프/전문 > 요리
├─ B종목:   번역                        ├─ B종목:   번역
├─ B급수:   교육 3급                     ├─ B급수:   전문 1급
├─ C사이트: 시험                         ├─ C사이트: 전문가 매칭
├─ D담당:   문제은행 팀 > AI번역          ├─ D담당:   강사 팀 > AI번역
└─ E역할:   강사 (조회+입력+편집)         └─ E역할:   외부 (조회+입력, 계약범위만)
```

## Part 4. 분류 <-> 시스템 매핑 (두 문서 연결)

분류표의 각 축이 기술 시스템의 어디에 구현되는지 매핑한다.

### 4-1. 분류축 → 시스템 컴포넌트

| 분류축 | 시스템 위치 | 역할 |
|--------|------------|------|
| **A 소재 분류** | `modules/data-definitions/` | 소재 트리(문서/음성/영상/...) 데이터를 JSON으로 정의. 단일 소스(Single Source of Truth) |
| **B 급수 체계** | `plugins/grade-selector/` | 5종목 x 3구분 x N급의 급수 선택 UI를 생성. 설정으로 종목별 분기 |
| **A x B 교차** | `plugins/field-selector/` | 소재 x 종목 조합 선택기. data-definitions의 트리 + grade-selector의 급수를 결합 |
| **C 홈페이지** | 별개 사이트 목록 (아래 매핑 참조) | 각 사이트가 플러그인을 embed |
| **D 담당 조직** | 향후 RBAC | D팀이 접근 **범위** 결정 (어느 테이블) |
| **E 역할** | 향후 RBAC | E역할이 접근 **동작** 결정 (조회/입력/편집/삭제/승인) |

### 4-2. C홈페이지 <-> 별개 사이트 매핑

| C홈페이지 분류 | 실제 사이트/시스템 | Work Studio 테이블 |
|---------------|-------------------|-------------------|
| 교육 | TESOL 랜딩 페이지 (`sites/tesol-landing/`) | `tesol_expert_applications`, `applications` |
| 번역 | AI번역 사이트 | `translation_*` (6개 테이블) |
| 통독 문서 | 통독v3 | `classic_inquiries`, `classic_specialist_apps` |
| 전시회 | 전시회/관광 사이트 | (향후 추가) |
| 전문가 매칭 | 전문가 매칭 사이트 | `translation_experts`, `aiethics_applications` |
| 시험 | CBT 시험 사이트 | `cbt_results`, `questions`, `level_test_submissions` |
| 그외 | 블로그 등 | `story_posts`, `applications` |

### 4-3. D담당 + E역할 → RBAC 설계

| 현재 상태 | 향후 계획 |
|-----------|----------|
| `admin.html`에 단일 API 키 인증 | 역할 기반 접근 제어 (RBAC) |
| 모든 관리자가 모든 테이블 접근 가능 | E역할 × D팀 조합으로 접근 범위 결정 |
| 역할 구분 없음 | 로그인 시 역할 판별 → 권한 매트릭스 적용 |

**RBAC 구현 시 흐름:**
```
로그인 → 역할 확인 (E: 강사) → 소속 팀 확인 (D: 강사팀-AI번역)
→ 접근 가능 테이블: AI번역 관련 (translation_*, tesol_expert_applications)
→ 허용 동작: 조회 + 입력 + 편집 (삭제 불가)
```

## Part 5. 모듈 & 플러그인

### 왜 필요한가?

> 수백 개 사이트에 신청폼을 하나하나 다 만들 수 없다. 한 번 만들고, 어디서든 꽂아 쓸 수 있어야 한다.

### 모듈 vs 플러그인

| | 모듈 (Module) | 플러그인 (Plugin) |
|---|---------------|-------------------|
| **비유** | 밀가루, 설탕, 이스트 (재료) | 즉석 빵 믹스 (물만 부으면 완성) |
| **누가 씀?** | 개발자 | 누구든 (비개발자도) |
| **어떻게 씀?** | 코드에서 import해서 조립 | script 태그 한 줄 복붙 |
| **새 사이트 추가** | 코드 작성 필요 | 설정(config)만 변경 |
| **업데이트** | 사이트마다 수동으로 | 중앙 1곳 고치면 전체 반영 |
| **CSS 커스텀** | 코드에서 수정 | 외부에서 덮어쓰기 가능 |
| **관계** | 플러그인의 **재료** | 모듈을 조합한 **완제품** |

### 사용 예시

**플러그인 (비개발자도 가능):**

<!-- TESOL 홈페이지에 급수 선택기 추가 -->

```js
import { fieldTree } from '../modules/data-definitions/fields.json';

```

## Part 7. 핵심 규칙 + 혼동 주의

### 핵심 규칙

| # | 규칙 | 이유 |
|---|------|------|
| 1 | 소비자 접수 기능을 AI Studio에 넣지 않는다 | AI Studio는 직원 전용. 소비자 데이터는 Work Studio |
| 2 | 공용 데이터 정의(분야/분류/급수)는 한 곳에만 | `modules/data-definitions/`가 단일 소스. 복사 금지 |
| 3 | 같은 기능을 2번 이상 만들면 플러그인으로 | 수백 개 사이트에서 쓸 거면 한 번만 만들고 설정으로 분기 |
| 4 | 새 사이트 추가 시 코드 최소화 | 플러그인 embed + CSS 커스터마이징으로 해결 |
| 5 | 별개 사이트의 백엔드는 Work Studio API 사용 | 데이터는 한 곳에 모인다. DB를 따로 만들지 않는다 |
| 6 | 플러그인은 `init()` 한 줄로 동작해야 | 호스트 사이트가 내부를 몰라도 되어야 진짜 플러그인 |

### 혼동 주의

| 헷갈리는 것 | 정리 |
|-------------|------|
| A열 이름이 "A분야"인데, 회의에서 "분야=프롬/번역/윤리" | A열 = **소재** 분류, 회의의 "분야" = B열의 **종목** |
| "대분류"가 A에도 B에도 있음 | A대분류 = 문서/음성/영상, B대분류 = 프롬/번역/윤리 → 서로 다른 축 |
| A에 "번역", "프롬프트"가 있고 B에도 "번역", "프롬"이 있음 | A = 작업 **방법론** 분류, B = 자격시험 **종목**. 이름만 같고 다른 것 |
| 급수 방향 | **숫자 클수록 높은 수준**. 교육1급(입문) < 교육8급 < 비즈니스 < 전문 |
| B의 "교육/비즈니스/전문"과 신청서의 "일반인/전문가/교육자" | B = **시험 난이도 구분**, 신청서 = **신청자 유형**. 완전히 별개 |
| A의 "확장영역" | 단순 구조가 아님. **도메인 스페셜리스트** 분류 (요리 전문 번역가 등) |
| "관리자가 쓰니까 AI Studio?" | **아니다**. 데이터 주체가 소비자/외부인이면 Work Studio |
| B급수 구분 명칭 "일반" | **"비즈니스"**가 정확한 명칭. 교육 / 비즈니스 / 전문 |
| E열이 "등급"이라 서열로 오해 | E는 **역할(Role)**. 개발≠팀장 위, 외부≠개발 위. 유형별 권한이 다를 뿐 서열 아님 |

                    ▼
            [Work Studio DB]   ←─── db.js (SQLite, JSON blob 스키마)
                    │                모든 사이트의 데이터가 여기에 모임
                    │
                    │ 관리자가 조회

            [Work Studio admin.html]     ← 소비자 데이터 관리

            [AI Studio]                  ← 직원/내부 업무 관리 (별도 DB)

    (Work Studio에서 직접 서빙하는 소비자 페이지)
            [/tesol-apply]               ← TESOL 신청 페이지
            [/level-test]                ← 레벨테스트 페이지
            [/form-engine/engine.js]     ← 외부 사이트 embed용 폼 엔진
